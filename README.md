# PromptSharp

PromptSharp is a prompt-optimization app with two usable entry points:

- `PromptSharp-static/` for a standalone browser-only version
- `frontend/` + `backend/` for the full app and API workflow

It focuses on turning rough prompts into cleaner, more structured instructions while applying configurable safety checks.

## Current Features

- Prompt optimization through OpenRouter models
- Searchable model selection with free-model prioritization
- Side-by-side prompt comparison
- Local prompt history with export support
- Guardrails for toxicity, PII, hallucination, truthfulness, and logical checks
- Browser-stored API key for the static and frontend experiences
- `Clear key` and `Clear local data` actions in both the static and frontend UIs

## Usage Options

### 1. Standalone Static Version

Use this when you want the quickest setup and do not want to run Node locally.

1. Open `PromptSharp-static/index.html` in your browser.
2. Enter your OpenRouter API key.
3. Paste a prompt and optimize it.
4. Use `Clear key` if you only want to remove the saved API key.
5. Use `Clear local data` if you want to wipe prompts, key, and local app state for that page.

Notes:

- This version runs entirely in the browser.
- Each user provides their own OpenRouter API key.
- Saved prompts and settings stay in browser storage only.

### 2. Full Frontend + Backend App

Use this when you want the repo's full app structure available locally or want to host frontend and backend separately.

## Local Setup

### Prerequisites

- Node.js 14+
- npm

### Backend

1. Go to the backend folder:

```bash
cd backend
```

2. Install dependencies:

```bash
npm install
```

3. Create `.env` from `.env.example` and set your real key:

```env
PORT=3000
OPENROUTER_API_KEY=your_openrouter_api_key
NODE_ENV=development
CORS_ORIGIN=http://localhost:5173
```

4. Start the backend:

```bash
npm run dev
```

The API will run on `http://localhost:3000`.

### Frontend

1. Go to the frontend folder:

```bash
cd frontend
```

2. Install dependencies:

```bash
npm install
```

3. Start the frontend:

```bash
npm run dev
```

4. Open:

```text
http://localhost:5173
```

Frontend notes:

- The UI stores the OpenRouter API key in browser storage.
- The current input flow passes that key directly for OpenRouter requests.
- `VITE_API_URL` is still supported in `frontend/src/services/api.js` if you want the frontend to call the backend API path instead.

## Guardrails

PromptSharp includes configurable checks for:

- Toxicity
- PII protection
- Hallucination detection
- Truthfulness verification
- Logical contradiction detection

Behavior:

- Unsafe or problematic prompts can be blocked before optimization.
- Detected issues are surfaced in the comparison UI.
- Guardrail state is stored with saved prompt history.

## Tests

The guardrails test runner is browser-based.

To run it:

1. Open `frontend/src/tests/guardrailsTestRunner.html` in your browser.
2. Click `Run All Tests`.

The tests cover harmful intent, hostile language, PII, and false-positive scenarios.

## Production Build

### Frontend build

Run:

```bash
cd frontend
npm run build
```

The Vite config writes the production output to the repo-root `dist/` folder.

### Backend

There is currently no separate backend build step. For production you run the backend with:

```bash
cd backend
npm start
```

## Hosting Notes

### Static hosting

The simplest public deployment is the standalone static package in `PromptSharp-static/`.

For Netlify Drop:

1. Open `https://app.netlify.com/drop`
2. Drag the contents of `PromptSharp-static/`, or just `PromptSharp-static/index.html`
3. Publish

### Split hosting

For the full app, a practical free-tier layout is:

- Netlify for the frontend
- Render for the backend

If deploying the frontend build output:

- Build command: `npm --prefix frontend install && npm --prefix frontend run build`
- Publish directory: `dist`

Set:

```env
VITE_API_URL=https://your-backend-host/api
```

## Security Notes

- Do not hardcode a real OpenRouter API key into checked-in HTML or frontend code.
- The static version is suitable only when users are expected to provide their own key.
- Backend TLS verification is enabled by default. `ALLOW_INSECURE_SSL=true` should only be used for local troubleshooting.

## Troubleshooting

- If optimization fails, verify your OpenRouter API key first.
- If the frontend cannot reach the backend, verify `VITE_API_URL` and backend health at `/api/health`.
- If model loading fails, refresh the model list and confirm the stored API key is valid.
- If local state seems stale, use `Clear local data`.
